
Below is a formal BRD / FRD-style system requirement specification for the Marketing Representative (MR) / Sales Agent Portal, written in clear, auditable, and implementation-ready language suitable for bank IT, compliance, and vendor evaluation.
________________________________________
Marketing Representative (MR) / Sales Agent Portal
Business Requirements & Functional Requirements
________________________________________
1. Business Objectives (BRD)
BRD-01
The system shall provide a secure digital portal to enable authorised Marketing Representatives (MRs) and Sales Agents to submit, track, and manage loan applications on behalf of customers.
BRD-02
The system shall improve loan application quality, reduce processing turnaround time, and minimise manual handling by Financial Institution (FI) officers.
BRD-03
The system shall support controlled bulk loan application submissions for eligible products and authorised MR groups while maintaining compliance and data integrity.
BRD-04
The system shall provide transparent communication and collaboration between MRs and FI officers with full auditability.
________________________________________
2. User Roles & Access Control (FRD)
FRD-01
The system shall support role-based access control (RBAC) for the following user roles:
•	Marketing Representative
•	Sales Agent
•	Agency Manager / Team Leader
•	FI Officer (Read / Review)
•	System Administrator
FRD-02
The system shall restrict access to loan products, submission modes, and data visibility based on user role and assigned permissions.
FRD-03
The system shall enforce secure authentication mechanisms, including multi-factor authentication (MFA) where applicable.
________________________________________
3. Loan Application Submission (FRD)
3.1 Single Loan Application (Primary Workflow)
FRD-04
The system shall allow MRs to submit loan applications individually via a guided digital workflow.
FRD-05
The system shall support saving loan applications in draft status prior to submission.
FRD-06
The system shall validate mandatory fields and required documents before allowing submission.
FRD-07
The system shall allow upload and management of supporting documents, including identity documents, income proof, and bank statements.
FRD-08
The system shall provide real-time validation feedback to the MR during data entry.
________________________________________
3.2 Bulk Loan Application Submission (Controlled Workflow)
FRD-09
The system shall support bulk loan application submission via structured file upload (e.g. Excel or CSV).
FRD-10
The system shall restrict bulk submission capability to authorised loan products and MR groups based on configurable business rules.
FRD-11
The system shall perform pre-processing validation on bulk upload files, including:
•	Mandatory field completeness
•	Data format validation
•	Duplicate record detection
FRD-12
The system shall allow partial acceptance of bulk submissions, whereby valid records are processed and invalid records are rejected.
FRD-13
The system shall generate a detailed error report for rejected bulk records.
________________________________________
4. Application Status Tracking (FRD)
FRD-14
The system shall allow MRs to view real-time loan application status.
FRD-15
The system shall display standardised application lifecycle stages, including:
•	Draft
•	Submitted
•	Under Review
•	Additional Information Required
•	Approved
•	Rejected
•	Cancelled
FRD-16
The system shall provide search and filtering capabilities based on application ID, customer name, status, and submission date.
FRD-17
The system shall display service-level indicators, including time spent in each processing stage.
________________________________________
5. Communication & Collaboration (FRD)
FRD-18
The system shall provide an internal secure messaging function for communication between MRs and FI officers.
FRD-19
The system shall link communications contextually to specific loan applications.
FRD-20
The system shall allow FI officers to request additional information or documents through the portal.
FRD-21
The system shall maintain a complete audit trail of all communications and document exchanges.
________________________________________
6. Document Management (FRD)
FRD-22
The system shall store all loan-related documents in a centralised repository linked to each application.
FRD-23
The system shall support document version control and status tagging (e.g. submitted, accepted, rejected).
FRD-24
The system shall enforce document access permissions based on user roles.
________________________________________
7. Notifications & Alerts (FRD)
FRD-25
The system shall generate in-portal notifications for key events, including status changes and information requests.
FRD-26
The system shall support configurable external notifications (e.g. email or push notifications) without exposing sensitive customer data.
________________________________________
8. Performance & Commission Monitoring (Optional / Configurable)
FRD-27
The system shall provide MRs with visibility into their loan application pipeline and performance metrics.
FRD-28
The system may support commission estimation and reporting, subject to contractual and regulatory constraints.
________________________________________
9. Compliance, Audit & Security (FRD)
FRD-29
The system shall log all user actions with timestamps, user IDs, and activity details.
FRD-30
The system shall support audit trail extraction for compliance and regulatory review.
FRD-31
The system shall comply with applicable data protection regulations, including PDPA requirements.
FRD-32
The system shall prevent unauthorised data export or sharing outside the approved channels.
________________________________________
10. System Integration (FRD)
FRD-33
The system shall integrate with the Loan Origination System (LOS) for application processing.
FRD-34
The system shall integrate with credit bureau and eKYC services where applicable.
FRD-35
The system shall support integration with document management and notification services.
________________________________________
11. Non-Functional Requirements (NFR)
NFR-01
The system shall support concurrent access by multiple MRs without performance degradation.
NFR-02
The system shall ensure high availability and data integrity.
NFR-03
The system shall provide configurable parameters to support new loan products and campaigns without code changes.
________________________________________
